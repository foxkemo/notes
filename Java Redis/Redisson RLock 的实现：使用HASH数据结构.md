# 1、Redisson 分布式锁用什么数据结构？

# 2、Redisson 锁的核心实现

以：

```java
RLock lock = redissonClient.getLock("myLock");
lock.lock();
```

为例👇

---

## 1️⃣ 底层存储结构

```bash
myLock → Hash
```

结构类似：

```text
myLock = {
   threadId1: 重入次数
}
```

👉 也就是说：

✔ 用的是 **Hash（哈希）结构**

---

## 2️⃣ 为什么用 Hash？

👉 为了支持：

### 🔥 可重入锁

```java
lock.lock();
lock.lock();
```

Redis 中：

```text
threadId → count++
```

---

# 四、加锁流程（核心）

## 👉 本质是 Lua 脚本（保证原子性）

逻辑：

```text
if (锁不存在) {
    创建锁（hash）
    设置过期时间
    返回成功
}
else if (是当前线程) {
    重入次数 +1
    返回成功
}
else {
    返回失败
}
```

---

# 五、解锁流程

```java
lock.unlock();
```

👉 Redis 做的事：

```text
重入次数 -1
如果 =0 → 删除 key
```

---

# 六、为什么不用 List / Set？

| 数据结构   | 能不能做锁 | 原因        |
| ------ | ----- | --------- |
| String | ✅     | 简单 key 判断 |
| Hash   | ✅     | 支持重入      |
| List   | ❌     | 无法标识线程    |
| Set    | ❌     | 无法计数      |

---

# 七、Redisson 的高级机制（加分🔥）

## 1️⃣ 看门狗（Watch Dog）

👉 自动续期：

```text
默认 30 秒过期
每 10 秒自动续命
```

防止：

> 业务没执行完锁就过期 ❌

---

## 2️⃣ Lua 脚本

👉 保证：

```text
加锁 / 解锁 = 原子操作
```

---

## 3️⃣ 发布订阅（Pub/Sub）

👉 等待锁时：

* 不是一直轮询
* 而是订阅解锁消息

# 八、面试总结（建议直接背）

👉

### 1️⃣ List 是否可以为空？

> Redis 中 List 不能存在“空结构”，当元素被移空后 key 会自动删除，所以空 List 等价于 key 不存在

---

### 2️⃣ Redisson 锁用什么结构？

> Redisson 分布式锁底层使用 Redis 的 Hash 结构存储线程 ID 和重入次数，并通过 Lua 脚本保证加锁和解锁的原子性，同时结合过期时间和看门狗机制实现可靠分布式锁
