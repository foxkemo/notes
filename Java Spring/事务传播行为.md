
在 Spring 事务管理中，当一个事务方法（外层）调用另一个事务方法（内层）时，其行为由 **事务传播行为（Propagation）** 决定。

这种情况最核心的知识点是：**内层事务如何影响外层事务，以及异常如何触发回滚。**

以下是三种最常见的情况解析：

---

### 1. `REQUIRED`（默认行为）：合并为一个事务
这是最常见的场景。内层事务发现已经存在一个事务，就会**加入**进去。

*   **行为**：内外层合用一个物理事务。
*   **回滚逻辑**：
    *   无论是内层还是外层抛出异常，**整个事务都会回滚**。
    *   **致命陷阱**：如果你在外部方法 `catch` 了内部方法的异常，但内部方法标记了回滚，你会收到 `UnexpectedRollbackException`。因为内层已经把事务标为 `RollbackOnly` 了，外层尝试 Commit 会失败。

### 2. `REQUIRES_NEW`：独立的新事务
内层事务会挂起外层事务，开启一个**全新、独立**的物理事务。

*   **行为**：两个事务互不干扰。
*   **回滚逻辑**：
    *   内层事务失败回滚，不影响外层事务（前提是外层捕获了内层的异常）。
    *   外层事务回滚，**不会**导致已经提交的内层事务回滚。
*   **适用场景**：**日志记录**。无论业务成功与否，都要把操作日志存入数据库。

### 3. `NESTED`：嵌套事务（基于 Savepoint）
这是真正的“嵌套”。它在当前事务中创建一个 **保存点（Savepoint）**。

*   **行为**：
    *   内层事务回滚，可以只回滚到保存点，**外层事务可以继续提交**。
    *   外层事务回滚，**内层事务必须跟着回滚**。
*   **前提**：仅支持 JDBC，且数据库驱动需要支持 Savepoint。

---

### 4. ⚠️ 必坑指南：自调用问题 (Self-Invocation)

这是开发中最容易犯的错。如果你在同一个类（Bean）中，方法 A 调用方法 B：

```java
@Service
public class CouponService {
    @Transactional
    public void methodA() {
        methodB(); // 这里的事务配置会失效！
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() { ... }
}
```

*   **现象**：`methodB` 的 `REQUIRES_NEW` 不会生效。
*   **原因**：Spring 的事务是基于 **AOP 代理** 的。当你直接调用同一个类的方法时，绕过了代理对象，Spring 拦截不到这个调用。
*   **解决方案**：
    1.  把 `methodB` 拆到另一个 Service 类中。
    2.  或者通过 `AopContext.currentProxy()` 获取当前代理对象再调用（不推荐）。
    3.  或者自己注入自己（Spring 支持循环依赖注入）。

---

### 5. 结合你之前的编程式事务

如果你在 `TransactionTemplate` 内部调用另一个 `@Transactional` 方法：

```java
transactionTemplate.executeWithoutResult(status -> {
    // 这里是外层事务（编程式）
    anotherService.methodB(); // 这里是内层事务（声明式）
});
```

*   **关系**：由于 `TransactionTemplate` 默认也是 `REQUIRED` 级别，`methodB` 会加入到 `transactionTemplate` 开启的这个事务中。
*   **异常处理**：如果在 `methodB` 报错，整个 `transactionTemplate` 块都会回滚。

### 总结建议

1.  **大多数情况用 `REQUIRED`**：保持原子性。
2.  **需要独立结果用 `REQUIRES_NEW`**：如流水号生成、审计日志。
3.  **处理局部失败用 `NESTED`**：如批量处理中，某一条数据失败不影响整体。
4.  **注意代理失效**：跨事务调用一定要跨类（不同 Bean 之间调用）。
