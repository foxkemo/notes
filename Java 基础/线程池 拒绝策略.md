当 `ExecutorService`（线程池）的任务队列已满，且工作线程也达到最大限制时，它就会进入“暴走”状态。为了防止系统崩溃，Java 提供了一套**拒绝策略（RejectedExecutionHandler）**。

虽然 Java 标准库内置了四种策略，你提到的“三种”可能是指最常用的那几类。我为你详细讲解最核心的三种，并附带一种作为补充。

---

## 1. AbortPolicy（直接抛出异常）

这是线程池的**默认策略**。它的逻辑非常简单粗暴：不干活，且直接报错。

- **行为：** 丢弃任务，并抛出 `RejectedExecutionException` 运行时异常。
    
- **适用场景：** 适用于对关键业务比较敏感的场景，你需要立即知道任务失败了，并由调用者负责捕获异常并进行后续处理。
    
- **缺点：** 如果不处理异常，可能会导致程序中断。
    

## 2. CallerRunsPolicy（调用者运行）

这是一种典型的**“降级”和“反压（Backpressure）”**策略。它不会丢弃任务，也不会报错。

- **行为：** 既然线程池忙不过来，那就让**提交任务的那个线程**（通常是主线程）亲自去执行这个任务。
    
- **适用场景：** 适用于不允许任务丢失、且希望通过降低任务提交速度来缓解线程池压力的场景。
    
- **优点：** 任务一定会被执行；同时由于主线程去干活了，它就没空提交新任务，自然起到了限流作用。
    

## 3. DiscardOldestPolicy（丢弃最老任务）

这是一种“喜新厌旧”的策略。

- **行为：** 线程池会悄悄丢弃队列中**存活时间最长（最靠前）**的任务，然后尝试把当前的新任务再次提交给线程池。
    
- **适用场景：** 适用于那些**具有时效性**的任务。比如实时监控数据，旧的数据丢了就丢了，保证最新的数据能存进去更重要。
    

---

## 总结对比

|**策略名称**|**行为特征**|**风险点**|
|---|---|---|
|**AbortPolicy**|抛异常报错|如果不捕获异常，系统可能崩溃|
|**CallerRunsPolicy**|谁提交谁执行|会阻塞提交线程，降低系统吞吐量|
|**DiscardOldestPolicy**|丢弃队列头部的任务|任务会悄无声息地丢失，无从追溯|

> **补充：** 其实还有第四种 **DiscardPolicy（直接丢弃）**，它比 `DiscardOldestPolicy` 更狠，是直接把当前提交的任务丢掉，且不发任何通知。这通常只用于那些无关紧要的日志记录或统计。
