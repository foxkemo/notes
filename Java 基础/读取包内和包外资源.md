
在 Java 开发中，无论是 `META-INF` 还是普通的静态资源，在打包成 JAR 后，它们在逻辑上都处于 **Classpath（类路径）** 之下。

虽然物理位置有别，但在代码层面，我们主要通过 **ClassLoader** 或 **Spring 的资源抽象**来读取它们。

---

### 1. 使用标准 Java API 读取

这是最底层、通用的方式，适用于任何 Java 项目。

#### A. 读取 `META-INF`

`META-INF` 目录通常位于类路径的根部。

Java

```
// 1. 读取特定的资源文件（如 spring.factories）
InputStream is = MyClass.class.getClassLoader()
    .getResourceAsStream("META-INF/spring.factories");

// 2. 读取 MANIFEST.MF (标准做法)
Enumeration<URL> resources = MyClass.class.getClassLoader()
    .getResources("META-INF/MANIFEST.MF");
while (resources.hasMoreElements()) {
    URL url = resources.nextElement();
    // 遍历所有依赖包中的 MANIFEST
}
```

#### B. 读取其他静态资源

注意 **`Class.getResource`** 与 **`ClassLoader.getResource`** 的路径区别：

- **`MyClass.class.getResource("file.txt")`**：相对于 `MyClass` 所在的包路径。
    
- **`MyClass.class.getResource("/file.txt")`**：相对于 Source Root 根目录。
    
- **`classLoader.getResource("file.txt")`**：始终相对于 Source Root 根目录（开头不需要 `/`）。

### 2. 使用 Spring 的 `ResourceLoader` (推荐)

如果你在 Spring 环境中，使用 `ResourceLoader` 会更加优雅且强大，因为它支持通配符和各种协议。

Java

```
@Service
public class MyService {
    @Autowired
    private ResourceLoader resourceLoader;

    public void readFiles() throws IOException {
        // 读取 META-INF 资源
        Resource metaResource = resourceLoader.getResource("classpath:/META-INF/my-config.properties");
        
        // 读取 static 资源
        Resource staticResource = resourceLoader.getResource("classpath:/static/index.html");

        InputStream inputStream = staticResource.getInputStream();
        // 处理流...
    }
}
```

### 3. 核心区别：单资源 vs 多资源扫描

读取这两类资源时，最大的逻辑区别在于：**你是要“精准投喂”还是“全网搜索”？**

|**场景**|**读取策略**|**典型案例**|
|---|---|---|
|**读取自己的静态资源**|**直接定位**：路径通常是固定的（如 `static/logo.png`）。|加载前端模板、读取应用本地配置。|
|**读取 `META-INF`**|**全类路径扫描**：往往需要扫描所有 JAR 包中的同名文件。|Spring Boot 查找所有依赖中的 `spring.factories` 来实现插件化。|

#### 技巧：使用 `PathMatchingResourcePatternResolver`

如果你想一次性读取所有依赖包中 `META-INF` 下的某个配置，Spring 提供了一个强力工具：

Java

```
ResourcePatternResolver resolver = new PathMatchingResourcePatternResolver();
// 使用 classpath*: 前缀扫描所有 JAR 包
Resource[] resources = resolver.getResources("classpath*:META-INF/my-plugin.xml");
```

### 4. 避坑指南：为什么我的资源读不到？

1. **路径大小写：** Linux 环境下（也是大多数服务器环境）文件名是区分大小写的，`meta-inf` 和 `META-INF` 不同。
    
2. **根路径斜杠：**
    
    - 使用 `ClassLoader` 时，路径**不能**以 `/` 开头。
        
    - 使用 `Class.getResource` 读取根目录资源时，路径**必须**以 `/` 开头。
        
3. **构建工具过滤：** 检查你的 `pom.xml` 或 `build.gradle`。如果配置了 `resources` 过滤（Filtering），有些二进制文件（如图片或字体）可能会在打包时被损坏。
    
    - _解决方案：_ 在 Maven 中配置 `<nonFilteredFileExtensions>` 排除这些后缀。
        
4. **文件系统 vs JAR：** 永远不要使用 `new File("...")` 来读取包内资源。在开发阶段它可能有效，但打成 JAR 包后，资源不再是文件系统的独立文件，必须使用 `getInputStream()`。
    

**总结：**

对于包内类来说，`META-INF` 和 `static` 只是 **Classpath 下的两个不同文件夹**。只要路径写对，读取它们的 API 是完全一致的。`META-INF` 更多地被框架用来做“元数据发现”，而 `static` 更多地被用来做“业务数据展示”。
