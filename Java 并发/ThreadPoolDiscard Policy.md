在 Java 线程池（`ThreadPoolExecutor`）中，**拒绝策略（RejectedExecutionHandler）** 是当线程池处于“饱和状态”时，决定如何处理新提交任务的最后一道防线。

### 1. 触发时机
满足以下两个条件时，拒绝策略会被调用：
1.  线程池中的工作线程数已达到 `maximumPoolSize`。
2.  任务队列（BlockingQueue）已满。
3.  或者线程池已关闭（`isShutdown`）。

---

### 2. JDK 内置的四种策略

#### ① AbortPolicy (默认策略)
*   **行为**：直接抛出异常 `RejectedExecutionException`。
*   **特点**：**“宁为玉碎，不瓦全”**。它会中断业务流程，让你知道系统处理不过来了。
*   **适用场景**：必须保证任务被处理，如果处理不了，需要立刻报错让上游感知并进行处理。

#### ② CallerRunsPolicy (调用者运行策略)
*   **行为**：哪来的回哪去。谁提交的任务，就由谁（当前线程）来执行这个任务。
*   **特点**：**“降级处理 + 天然限流”**。
    *   它不会丢弃任务，也不会抛出异常。
    *   由于主线程（调用者线程）去执行任务了，它在忙碌期间就不能再往线程池提交新任务，从而自然而然地减慢了任务提交的速度。
*   **适用场景**：**业务非常重要，绝对不能丢失任务**，且允许主线程被阻塞（吞吐量下降）。

#### ③ DiscardPolicy (丢弃策略)
*   **行为**：默默地丢弃提交的任务，不报任何错误。
*   **特点**：**“佛系丢弃”**。你提交了任务，任务消失了，你却不知道。
*   **适用场景**：任务不重要，丢了就丢了（比如：实时日志收集、非关键性的状态上报）。
*   *注意：你在优惠券代码里用的就是这个，这在金融/资产类业务中通常是危险的。*

#### ④ DiscardOldestPolicy (丢弃最老策略)
*   **行为**：丢弃任务队列中最前面的（排队最久的）任务，然后重新尝试执行当前任务。
*   **特点**：**“喜新厌旧”**。保证最新的任务能被处理，旧的任务被抛弃。
*   **适用场景**：时效性要求高的任务（比如：股票行情信息，旧的数据不如新的有价值）。

---

### 3. 策略对比总结

| 策略 | 行为 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **AbortPolicy** | 抛异常 | 及时反馈，避免任务积压 | 业务会中断，需额外写 try-catch |
| **CallerRunsPolicy** | 调用者线程执行 | **不丢任务**，自带限流 | 可能会阻塞主线程（如 Tomcat 线程） |
| **DiscardPolicy** | 静默丢弃 | 性能开销最小 | **任务丢失无感知**，最危险 |
| **DiscardOldestPolicy** | 弃老取新 | 保证任务的“新鲜度” | 旧任务丢失，依然无明显反馈 |

### 4. 自定义拒绝策略 (Best Practice)
在生产环境中，我们往往不满足于以上四种。通常会实现 `RejectedExecutionHandler` 接口来打印关键日志、报警，或者将任务持久化到数据库。

```java
public class MyRejectedHandler implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 1. 打印 error 日志，记录任务详情
        log.error("线程池已满！任务丢弃，任务详情: {}", r.toString());
        // 2. 埋点监控，触发预警
        Metrics.counter("thread.pool.reject").increment();
        // 3. (可选) 尝试持久化到 Redis/DB 稍后补偿执行
    }
}
```

### 5. 针对你之前“优惠券任务”代码的建议

你的代码：
```java
new ThreadPoolExecutor.DiscardPolicy()
```
**诊断：**
优惠券发放是非常核心的资产操作。如果因为瞬间并发高，线程池满了，系统会**静默丢弃**发放任务。结果就是：**日志显示任务已创建，但用户永远收不到券，且没有任何错误日志。** 排查起来非常痛苦。

**建议改进：**
1.  **首选 `CallerRunsPolicy`**：如果发放任务满了，让 Web 线程自己发。虽然接口响应变慢，但至少券发出去了。
2.  **次选 `AbortPolicy`**：抛出异常，让 Spring 事务回滚，给前端返回“系统繁忙”，让用户重试。
3.  **持久化**：在 `createCouponTask` 之前先存库（状态为“待发送”），即使线程池满了，也可以通过定时任务扫描数据库把漏掉的券补发出去。
