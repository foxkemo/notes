除了不能使用显式事务控制（`COMMIT`/`ROLLBACK`）外，MySQL 的 **Function (自定义函数)** 相比于 **Procedure (存储过程)** 还有许多严格的限制。这些限制主要是为了保证函数在被 `SELECT` 等语句调用时的**确定性**和**安全性**。

以下是 Function 的主要限制：

---

## 1. 语句类型的限制

函数内部**禁止执行 DDL 语句**和某些特定的 SQL 语句：

- **禁止 DDL：** 你不能在函数里执行 `CREATE TABLE`、`ALTER TABLE`、`DROP TABLE` 或 `TRUNCATE`。
    
- **禁止显式事务：** 无法使用 `START TRANSACTION`、`SAVEPOINT` 等。
    
- **禁止动态 SQL：** 函数内不能使用 `PREPARE` 和 `EXECUTE` 运行动态生成的字符串 SQL。
    

---

## 2. 返回值的强制性

- **必须有返回值：** 函数体中必须包含至少一个 `RETURN` 语句。
    
- **单一返回值：** 只能返回一个值（如 `INT`, `VARCHAR` 等），不能像存储过程那样通过 `OUT` 参数返回多个值，也不能直接返回一个结果集（Result Set）。
    

---

## 3. 调用上下文的限制（Side Effects）

如果你在 `SELECT` 语句中调用函数，该函数会受到更严苛的限制：

- **不能修改表数据：** 如果函数是在 `SELECT` 语句中被调用的，那么这个函数内部**不允许**执行 `INSERT`、`UPDATE` 或 `DELETE`。
    
    - _例外：_ 如果你是通过 `SET @var = my_func()` 这种独立方式调用，函数内部是可以改数据的，但这通常被视为不良设计。
        
- **递归限制：** MySQL 默认不支持函数的递归调用（虽然可以通过配置更改，但极度不推荐，容易导致堆栈溢出）。
    

---

## 4. 确定性声明 (Characteristic)

在创建函数时，你必须声明它的**特性**，否则在开启了二进制日志（Binary Logging）的环境下会报错。

- **`DETERMINISTIC` (确定性)：** 输入相同参数，永远返回相同结果。
    
- **`NOT DETERMINISTIC`：** 结果可能随时间或随机数改变。
    
- **`READS SQL DATA` / `MODIFIES SQL DATA`：** 告诉 MySQL 这个函数是否涉及数据库读写。
    

---

## 5. 参数与结果集

- **仅支持 `IN` 参数：** 函数的参数默认为输入类型，不支持存储过程那样的 `OUT`（输出）或 `INOUT`（输入输出）模式。
    
- **不能返回结果集：** 函数不能执行一个 `SELECT * FROM table` 并把这些行直接“吐”给调用者。如果需要处理多行数据，必须在函数内部使用 **游标 (Cursor)**。
    

---

## 6. 与存储过程的区别总结

|**限制维度**|**Function (函数)**|**Procedure (存储过程)**|
|---|---|---|
|**事务 (Commit/Rollback)**|**严禁使用**|**完全支持**|
|**被 SELECT 调用**|支持|不支持 (必须用 `CALL`)|
|**修改表数据**|在 `SELECT` 中调用时禁止修改|随时可以修改|
|**动态 SQL (Prepare)**|不支持|支持|
|**返回结果集**|不支持|支持|

---

### 💡 核心建议

在你的 **Java 后端开发** 实践中：

- 如果逻辑涉及 **写操作**（INSERT/UPDATE）或 **多步事务**，请务必使用 **Procedure** 或在 **Java Service 层** 控制。
    
- 如果逻辑仅仅是 **数据转换**（如：脱敏手机号、计算两个经纬度的距离、格式化复杂的字符串），使用 **Function** 会非常方便，因为它能直接写在 SQL 语句里，减少了数据在网络间的传输。
