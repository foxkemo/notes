这是一个非常关键的误区，我们需要从**“当前事务”**和**“外部事务”**两个视角来拆解。

结论是：**在持有 S 锁的情况下可以改，但极度不推荐。**

---

## 1. 视角一：当前事务（持有锁的人）

如果你在同一个事务内，先执行了 `FOR SHARE`，紧接着执行 `UPDATE`，**MySQL 会允许你修改**。

- **逻辑表现：** MySQL 会尝试将你手中的 **S 锁（共享锁）** 升级为 **X 锁（排他锁）**。
    
- **前提条件：** 此时**没有其他任何事务**也持有这一行的 S 锁。
    
- **结果：** 升级成功，数据修改完成。
    

---

## 2. 视角二：外部事务（其他人）

如果你持有 S 锁，**其他任何事务**都绝对无法修改这行数据。

- **原因：** 修改数据需要获取 X 锁，而 X 锁与你持有的 S 锁是**互斥**的。
    
- **表现：** 别人的 `UPDATE` 语句会一直卡在那（Lock Wait），直到你提交或回滚事务释放了 S 锁。
    

---

## 3. 为什么大家都说“S 锁不可改”？

虽然技术上允许“S 转 X”，但在**业务代码（Java/Go等）**中，这被视为一种**自杀式写法**。

### 致命的“死锁”逻辑：

想象你的短链接统计业务，两个请求（线程）同时进来：

1. **线程 A**：执行 `SELECT ... FOR SHARE`（拿到 S 锁）。
    
2. **线程 B**：执行 `SELECT ... FOR SHARE`（拿到 S 锁，S 锁互相兼容，所以 B 也能拿到）。
    
3. **线程 A**：执行 `UPDATE`（尝试升级为 X 锁，但发现 B 拿着 S 锁，A 开始**等待** B 释放）。
    
4. **线程 B**：执行 `UPDATE`（尝试升级为 X 锁，但发现 A 拿着 S 锁，B 开始**等待** A 释放）。
    

**结果：** 两个线程永久死锁。MySQL 检测到后，会报错并强制回滚其中一个。

---

## 4. 业务中的正确姿势

既然 S 锁转 X 锁这么危险，你应该怎么选？

### 如果你的目的是“读完就改”：

**绝对不要用 S 锁 (`FOR SHARE`)。**

直接使用 `FOR UPDATE`。

- **流程：** 线程 A 拿到 X 锁 -> 线程 B 想拿锁直接被阻塞在第一步 -> 线程 A 修改完提交 -> 线程 B 拿到锁开始工作。
    
- **优点：** 顺序排队，**永不死锁**。
    

### 如果你的目的是“只读不改”：

**使用 S 锁 (`FOR SHARE`)。**

- **场景：** 你在生成一份复杂的链路分析报表，需要确保读取的配置项在分析过程中不被后台管理员修改。
    

### 如果只是为了增加计数：

**连查询都省了。**

直接执行：

SQL

```
UPDATE link_statistics SET click_count = click_count + 1 WHERE link_id = 123;
```

- **原理：** MySQL 会在执行这一行时自动加上行级 X 锁，速度最快，冲突最少。
    

**一句话总结：** S 锁是为了**约束别人**不准改，而不是为了**方便自己**改。如果你想改，请直接申请 X 锁（`FOR UPDATE`）。
