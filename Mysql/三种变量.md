在数据库开发（以 MySQL 为例）中，变量主要分为三类：**局部变量**、**用户变量**和**系统变量**。

以下是它们的详细对比总结：

---

### 1. 局部变量 (Local Variables)
这种变量最像 Java 里的方法内部变量，仅在存储过程或函数内部有效。

*   **声明方式**：必须使用 `DECLARE` 关键字，且必须放在 `BEGIN...END` 块的最开头。
*   **前缀**：**无**前缀（直接写名字）。
*   **作用域**：仅限当前的 `BEGIN...END` 代码块。一旦过程执行结束，变量立即销毁。
*   **典型用法**：
    ```sql
    DECLARE my_count INT DEFAULT 0;
    SET my_count = 10;
    ```

### 2. 用户变量 (User-defined Variables)
这种变量是“会话级”的，只要你的数据库连接（Session）没断开，它的值就一直存在。

*   **声明方式**：**无需声明**，直接赋值即创建。
*   **前缀**：必须带 **`@`** 符号。
*   **作用域**：整个当前连接（Session）。在存储过程 A 里赋值，在存储过程 B 里也能读到。
*   **典型用法**：
    ```sql
    SET @user_name = '张三';
    SELECT @user_name; -- 在任何地方都能读取
    ```

### 3. 系统变量 (System Variables)
这种变量由数据库服务器维护，用来控制数据库的运行特性（如编码、最大连接数、自动提交等）。

*   **声明方式**：由系统预定义，用户只能修改（有权限的话），不能创建。
*   **前缀**：必须带 **`@@`** 符号（部分可以直接通过名字访问）。
*   **作用域**：
    *   **GLOBAL**（全局）：影响整个数据库实例，重启可能失效（除非修改配置文件）。
    *   **SESSION**（会话）：仅影响当前连接。
*   **典型用法**：
    ```sql
    SELECT @@max_connections;    -- 读取全局最大连接数
    SET @@autocommit = 0;        -- 关闭当前会话的自动提交
    ```

---

### 快速对比表

| 特性 | 局部变量 | 用户变量 | 系统变量 |
| :--- | :--- | :--- | :--- |
| **关键字/前缀** | `DECLARE` (无前缀) | `@` | `@@` |
| **是否需显式声明** | 是 | 否 | 系统预定义 |
| **存活时间** | 过程/函数结束即销毁 | 连接断开即销毁 | 随服务器运行或连接存在 |
| **使用位置** | 只能在 `BEGIN...END` 内部 | 任何 SQL 语句中 | 任何 SQL 语句中 |
| **Java 对应概念** | 方法局部变量 | 静态 Map 里的 Session 数据 | 系统环境变量 / 配置文件 |

---

### ⚠️ 开发建议（结合分布式场景）

1.  **分库分表环境下**：严禁依赖**用户变量 (`@`)**。因为在连接池模式下，同一个逻辑事务的不同 SQL 可能会切换物理连接，导致 `@` 变量丢失或读到错误的值。
2.  **存储过程逻辑**：优先使用**局部变量 (`DECLARE`)**，因为它们相互隔离，更安全。
3.  **参数传递**：如果需要将外部值传进存储过程，**永远使用参数 (IN/OUT)**，这是最稳健的做法。
