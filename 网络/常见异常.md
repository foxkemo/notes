在 Spring Boot Web 开发中，处理请求时常见的异常主要可以分为两大类：**反序列化异常**（通常与 JSON/XML 解析相关）和 **参数与方法不合法异常**（与路由、校验、绑定相关）。

以下是这些异常的详细列举及其触发场景：

---

### 一、 反序列化相关异常 (Deserialization Exceptions)
这类异常通常发生在 Spring 使用 `HttpMessageConverter`（如 Jackson）将请求体（`@RequestBody`）转换为 Java 对象时。

1.  **`HttpMessageNotReadableException`**
    *   **原因**：这是最常见的反序列化根异常。
    *   **场景**：
        *   请求体为空，但后端要求必填。
        *   JSON 格式错误（如缺少引号、逗号、括号不匹配）。
        *   后端字段是 `Enum`，但前端传了不存在的枚举值。
2.  **`InvalidFormatException`** (Jackson 抛出)
    *   **原因**：值类型不匹配。
    *   **场景**：前端传了 `"abc"`，但后端实体类对应字段是 `Integer` 或 `Date` 且无法解析。
3.  **`UnrecognizedPropertyException`**
    *   **原因**：JSON 包含未知字段。
    *   **场景**：前端传了 `{"age": 18, "unknown": "xxx"}`，但后端 DTO 只有 `age` 字段（且未配置 `ignoreUnknown = true`）。
4.  **`JsonParseException`**
    *   **原因**：JSON 语法语法错误。
    *   **场景**：请求体不是标准的 JSON 字符串。
5.  **`JsonMappingException`**
    *   **原因**：JSON 结构与 Java 对象结构不匹配。
    *   **场景**：前端传了一个数组 `[]`，后端却是一个对象 `{}`。

---

### 二、 参数不合法异常 (Parameter Binding Exceptions)
这类异常通常发生在参数绑定阶段，即 Spring 尝试将 URL 参数、路径变量或表单数据映射到方法参数时。

1.  **`MethodArgumentTypeMismatchException`**
    *   **原因**：方法参数类型转换失败。
    *   **场景**：`@PathVariable` 或 `@RequestParam` 定义为 `Long id`，但请求 URL 是 `/user/abc`（"abc" 无法转为 Long）。
2.  **`MissingServletRequestParameterException`**
    *   **原因**：缺少必选的查询参数。
    *   **场景**：方法定义了 `@RequestParam String name`，但请求 URL 中没有 `?name=xxx` 且 `required` 默认为 true。
3.  **`MissingPathVariableException`**
    *   **原因**：缺少路径变量。
    *   **场景**：`@GetMapping("/{id}")` 但实际配置或请求路径未能正确解析到该变量。
4.  **`ServletRequestBindingException`**
    *   **原因**：请求绑定相关的其他异常（是上述几个异常的父类）。
    *   **场景**：缺少必要的 Header（`@RequestHeader`）或 Cookie。

---

### 三、 数据校验异常 (Validation Exceptions)
当使用 Hibernate Validator（`@Valid`, `@Validated`）时抛出。

1.  **`MethodArgumentNotValidException`**
    *   **原因**：`@RequestBody` 参数校验失败。
    *   **场景**：实体类字段上有 `@NotNull` 或 `@Max`，前端传的值不符合要求。
2.  **`BindException`**
    *   **原因**：表单提交（`application/x-www-form-urlencoded`）或 GET 请求的对象参数校验失败。
    *   **场景**：非 JSON 提交时，对象数据绑定出错。
3.  **`ConstraintViolationException`**
    *   **原因**：单参数校验失败（通常用于 Service 层或在 Controller 方法参数上直接写注解）。
    *   **场景**：在 Controller 上加了 `@Validated`，并在方法参数前写 `(@Min(1) Long id)`。

---

### 四、 HTTP 请求/方法不合法异常
这类异常通常与接口定义本身不匹配有关。

1.  **`HttpRequestMethodNotSupportedException`**
    *   **原因**：HTTP 方法不支持（HTTP 405）。
    *   **场景**：接口定义为 `@PostMapping`，但前端使用了 `GET` 访问。
2.  **`HttpMediaTypeNotSupportedException`**
    *   **原因**：不支持的媒体类型（HTTP 415）。
    *   **场景**：后端要求 `Content-Type: application/json`，但前端发送的是 `text/plain` 或 `application/x-www-form-urlencoded`。
3.  **`HttpMediaTypeNotAcceptableException`**
    *   **原因**：Accept 头部冲突（HTTP 406）。
    *   **场景**：客户端要求返回 `application/xml`，但服务器只能提供 `application/json`。
4.  **`NoHandlerFoundException`**
    *   **原因**：找不到对应的 Handle（HTTP 404）。
    *   **场景**：访问了不存在的路径（注：Spring Boot 默认会跳转到 /error 页面，需配置才会抛出此异常）。

---

### 总结与建议方案

为了统一处理这些异常，建议在项目中创建一个 **`@RestControllerAdvice`** 类进行全局捕获，返回标准的错误结构：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 1. 处理反序列化及请求体错误 (400)
    @ExceptionHandler(HttpMessageNotReadableException.class)
    public Response handleHttpMessageNotReadableException(HttpMessageNotReadableException e) {
        return Response.fail(400, "请求体格式错误或读取失败");
    }

    // 2. 处理方法参数类型不匹配 (400)
    @ExceptionHandler(MethodArgumentTypeMismatchException.class)
    public Response handleTypeMismatch(MethodArgumentTypeMismatchException e) {
        return Response.fail(400, String.format("参数类型不匹配: %s", e.getName()));
    }

    // 3. 处理校验异常 (JSON提交) (400)
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Response handleValidationException(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldError().getDefaultMessage();
        return Response.fail(400, msg);
    }

    // 4. 处理 HTTP 方法不支持 (405)
    @ExceptionHandler(HttpRequestMethodNotSupportedException.class)
    public Response handleMethodNotSupported(HttpRequestMethodNotSupportedException e) {
        return Response.fail(405, "不支持的请求方法: " + e.getMethod());
    }
    
    // 5. 处理缺少必要参数 (400)
    @ExceptionHandler(MissingServletRequestParameterException.class)
    public Response handleMissingParam(MissingServletRequestParameterException e) {
        return Response.fail(400, "缺少必要参数: " + e.getParameterName());
    }
}
```

通过这种方式，可以将原本混乱的服务器堆栈信息，转化为友好的前端提示。
